通过前面的步骤，我们已经在openshift上部署好了我们的测试项目，并提供了redis服务，我们可以在界面上看到启动的效果。

从图中可以看到，我们分别启动了redis和docker-demo的应用，并通过动态扩展功能将docker-demo应用扩展到3个。

直接对Pod进行访问

每个Pod都在以Docker Container的方式启动运行，都会有自己的IP地址等信息，我们可以只是使用这些信息来与pod进行访问交互。

访问并测试redis的Pod

我们可以在浏览器控制台左侧的 Applications ▸ Pods ▸ redis-n-xxxx 来查看redis pod的详细信息。

图 1. redis pod的详细信息
从图中的红框中我们可以看到当前pod运行的IP地址 172.17.0.3 和服务端口为 6379，我们可以通过redis client工具来进行连接。

使用nc工具来进行访问测试，执行命令：

$ nc 172.17.0.3 6379
图 2. redis访问测试结果
通过nc工具来与redis服务建立连接，并通过 get 和 set 命令来完成简单的数据存取操作。

访问并测试demo的Pod

我们可以在浏览器控制台左侧的 Applications ▸ Pods ▸ docker-demo-n-xxxx 来查看demo pod的详细信息。

图 3. demo pod的详细信息
从图中的红框中我们可以看到当前pod运行的IP地址 172.17.0.2 和服务端口为 3000，我们可以通过curl工具来进行访问。

使用curl工具来访问测试，执行命令：

$ curl 172.17.0.2:3000

我们可以获取到服务的返回结果：{"helloWorld":访问次数}

通过Service的方式访问应用

由于在运行中，pod会根据我们的需求进行动态扩展和销毁，而pod的ip也是在动态变化的，所以如果我们在应用中使用了pod的IP地址来进行连接，可能在pod重启后将会造成无法连接。 所以在Application创建的时候会响应的创建一个Service，Service也具有自己的IP地址等信息，但是Service是和Application进行绑定的，只有在Application被删除时才会进行失效，而通过Service来访问 可以进行负载均衡访问。

我们可以在浏览器 控制台左侧菜单 ▸ Application ▸ Services 来查看Services的列表信息。

图 4. Services列表

通过Service访问redis应用

从上图我们可以看到 redis 服务的 Cluster IP 为 172.30.12.167 ，我们重复前面的验证方式，将IP地址换为Service的IP地址。

使用nc工具来进行访问测试，执行命令：

$ nc 172.30.12.167 6379
图 5. redis测试访问结果
我们直接使用 get 指令来获取我们前面存储的内容。

通过Service访问demo应用

从上图我们可以看到 docker-demo 服务的 Cluster IP 为 172.30.78.110 ，我们重复前面的验证方式，将IP地址换为Service的IP地址。

使用curl工具来访问测试，执行命令：

$ curl 172.30.78.110:3000
图 6. demo测试访问结果
我们同样可以获取响应结果，如果一直重复访问，可以看到访问次数会一直增加。

通过Service的域名来访问应用

我们前面在 DNS搭建配置 中介绍过，使用dns我们可以通过默认的域名规则来访问我们的应用。

首先我们按照默认规则，生成redis Service的访问域名：redis.mytest.svc.cluster.local，执行命令：

$ dig redis.mytest.svc.cluster.local
图 7. 域名解析结果
在 dig 的查询信息中可以看到有一条 ANSWER ，响应的信息中有个A记录的IP，刚好就是我们的Service ClusterIP地址 172.30.12.167。

同样的，我们使用 nc 工具来进行redis服务测试，执行命令：

$ nc redis.mytest.svc.cluster.local 6379
图 8. redis测试访问结果
我们直接使用 get 指令来获取我们前面存储的内容。

同样的，使用 url 工具来进行docker-demo服务测试，执行命令：

$ curl docker-demo.mytest.svc.cluster.local:3000
图 9. demo测试访问结果
我们同样可以获取响应结果，如果一直重复访问，可以看到访问次数会一直增加。
