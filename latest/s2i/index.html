S2I示意图

Openshift 使用 S2I 支持从项目源码或者产品发布包直接进行编译或部署。

图 1. S2I原理示意图
S2I流程中使用的Image为定制后的Image，该Image包含了项目的运行时必须的环境和项目代码的编译处理过程。在 github/sclorg 上已经提供了一些常用的Container示例项目。 如果我们需要基于自己的产品来做环境，则需要进行定制对应的S2I Container。

S2I下载与安装

Sourece To Image(S2I、STI)是一个开源项目，项目代码共享在 github/openshift 中，大家可以从 releases 列表中下载相应的包。 文档中我们主要使用的版本为source-to-image-v1.1.2-5732fdd-linux-amd64.tar.gz。

下载好相应的发布包后，我们直接解压使用即可使用。

S2I介绍

S2I是一个构建的框架，主要是通过定制生成Docker Image。而定制后的Docker Image包含了必要的运行环境，可以直接对项目的源码或发布包进行部署打包，发布成新的Docker Image提交给openshift使用。

制作TongWeb的S2I Container

由于我们的Web App的项目要以TongWeb作为运行环境，要在Openshift上实现通过项目的源码进行自动构建部署。这里我们主要分为两个环节，第一个是准备示例项目的代码，我们已经创建好了一个示例项目， 第二个就是准备我们的TongWeb环境的Docker Image，来作为项目运行时的容器，创建Docker Image我们可以使用Dockerfile来进行创建，由于我们需要直接通过项目源码来进行部署，所以需要使用S2I来创建一个Dockerfile。

创建构建项目

输入命令：

$ s2i create boyingking/tongweb s2i-tongweb (1) (2)
1 s2i工具的create指令，具体的使用方式可以查看官方介绍
2 boyingking/tongweb 是一个已经制作好的Docker Image，可以通过docker run来启动运行，其中包含了TongWeb6的环境，镜像已经发布在Docker Hub

执行完成后，会在当前目录创建一个以s2i-tongweb为名的目录

图 3. 生成的目录结构

这是一个s2i标准的项目结构，其中我们主要关注一下几项：

Dockerfile

标准的Dockerfile文档，主要负责生成Docker Image
Makefile

make的编译脚本，描述了将会如何去编译我们的项目，文档中主要调用了Docker的docker build来完成Image的创建。
.s2i/bin

该目录中的几个文件是标准的Shell脚本，被称作 S2I Scripts ，根据文件名我们可以看出主要会在项目源码编译部署时使用。
test/test-app

该目录我们可以在制作过程中放一下测试的项目进行打包测试工作。
关于S2I Scripts的详细介绍可以参考官方介绍文档

定制脚本

我们创建好项目之后，需要按照我们的要求对一些文件进行修改。我们要实现的就是基于 boyingking/tongweb 来完成一个可以支持openshift进行自动部署的Container。

Step 1 修改Dockerfile

由于Dockerfile文件过长，这里只截取了我们比较关注的项，未列出来的保持默认就可以了。

Dockerfile
# boyingking/tongweb
FROM boyingking/tongweb:6.0 (1)

# TODO: Rename the builder environment variable to inform users about application yo
u provide them
ENV BUILDER_VERSION 6.0 (2)

# TODO: Set labels used in OpenShift to describe the builder image (3)
LABEL io.k8s.description="Platform for building tongweb" \
      io.k8s.display-name="builder 6.0" \
      io.openshift.expose-services="8080:http 9060:manager" \
      io.openshift.tags="builder,tongweb,6.0"

LABEL io.openshift.s2i.scripts-url=image:///usr/libexec/s2i (4)
COPY ./.s2i/bin/ /usr/libexec/s2i

RUN chown -R 1001:root /opt  (5)
RUN chown -R 1001:root /tmp
RUN chown -R 1001:root /home/tw6/tongweb6

EXPOSE 8080 8043 9060 (6)

CMD ["usage"] (7)
1 修改 FROM 为我们的TongWeb6的Docker Image。
2 设置一些环境变量，我们就测试性的设置一个版本为基于TongWeb6.0的制作。
3 设置一些标签，可以在Openshift中展示使用。
4 这个很重要，需要设置 io.openshift.s2i.scripts-url 的值，不然会找不到S2I Scripts的位置。并通过 COPY 将当前的.s2i/bin下的脚本复制到容器中。
5 进行一些目录的权限设置。
6 设置对外服务的端口列表。
7 执行一些其他的脚本。

Step 2 修改assemble脚本

assemble脚本主要负责源码的打包处理，我们可以在这里通过脚本完成对项目源码的处理，比如编译、打包、部署。

assemble
# Restore artifacts from the previous build (if they exist).
#
#if [ "$(ls /tmp/artifacts/ 2>/dev/null)" ]; then   (1)
#  echo "---> Restoring build artifacts..."
#  mv /tmp/artifacts/. ./
#fi

shopt -s dotglob
echo "---> Installing application source..."
mkdir /home/tw6/tongweb6/autodeploy/app (2)
mv /tmp/src/* /home/tw6/tongweb6/autodeploy/app (3)
1 注释掉一些不用的脚本，可以留着，这里我就注释了，避免一些错误。
2 由于TongWeb的应用限制，项目部署必须在autodeploy中以目录的形式存在，而我们拿到的是源码，所以默认创建了应用名为：app。
3 将要部署的项目代码移动到TongWeb容器目录中。由于我们的测试项目是静态jsp的，所以可以直接移动过去。

Step 3 修改run脚本

run脚本主要负责启动后执行的一些命令。

assemble
#!/bin/bash -e
#
# S2I run script for the 'tweb' image.
# The run script executes the server that runs your application.
#
# For more information see the documentation:
#	https://github.com/openshift/source-to-image/blob/master/docs/builder_image.md
#

exec /home/tw6/tongweb6/bin/startserver.sh (1)
1 修改执行的命令为TongWeb的启动命令。

编译成Image

在项目的根目中执行命令：

$ make (1)
1 该命令会执行Makefile中定义的操作逻辑。编译完成后，我们可以通过命令docker images看到本地仓库中的已经编译好的Docker Image了。
编译完成后会在本地的Docker仓库中，openshift在使用的时候需要从在线仓库中进行校验，而我们在前面也搭建了自己的私有仓库，所以我们需要将编译好的Docker Image推送到私有仓库中， 具体的推动方法可以查看将本地镜像Push到私有仓库中。

测试S2I打包

我们可以通过s2i命令来编译我们的测试项目，从而生成可以直接由Docker Run的Image。这里我们将前面准备好的示例项目代码放到了test/test-app目录中，然后执行命令：

$ s2i build test/test-app/ s2i-tongweb my-test-app (1)
1 使用 build 参数来进行编辑，格式为：build 源码目录 前面定制的支持s2i的image 新的image的名称。

执行完成后，我们可以通过docker images看到有新生成的image，名字为：my-test-app。

接着我们可以通过docker rum my-test-app来启动容器，可以测试我们的项目是否已经部署运行。
